Reverse Engineering C Programs

C is low-level enough to feel like you're holding a scalpel.

Example C Program
C
#include <stdio.h>
int factorial(int n) {
if(n <= 1) return 1;
return n * factorial(n-1);
}
int main() {
int x;
printf("Enter a number: ");
scanf("%d", &x);
printf("Factorial: %d\n", factorial(x));
return 0;
}

Compile:

gcc -o factorial factorial.c

Binary now contains:

  • .text ‚Äì code
  • .data ‚Äì initialized variables
  • .bss ‚Äì uninitialized variables

Disassembling:

objdump -d factorial

You might see:

call factorial
mov eax, edi
imul eax, eax
ret

Reverse engineering C requires:

  • Following stack usage
  • Recognizing function prologues/epilogues
  • Understanding memory addresses

Reverse Engineering C++ ‚Äì The Archaeology

C++ adds classes, inheritance, and vtables.

Example C++ Program
C++
#include <iostream>
class User {
public:
int id;
bool isAdmin;
void printStatus() {
if(isAdmin)
std::cout << "Admin\n";
else
std::cout << "User\n";
}
};
int main() {
User u{1, true};
u.printStatus();
return 0;
}

After compilation:

  • Classes disappear
  • Methods are linked via vtables
  • Names are often mangled: _ZN4User11printStatusEv

Reverse Engineering Tips:

  • Identify vtables
  • Track memory offsets
  • Reconstruct class structure
Even without names, memory layout shows:
‚Ä¢ id at offset 0
‚Ä¢ isAdmin at offset 4

Reverse Engineering Java ‚Äì The Talkative One

Java compiles to bytecode (.class) that retains structure:

Example Java Code
Java
public class Login {
public boolean check(String password) {
return password.equals("secret");
}
}

Decompile with Jadx / CFR:

jadx-gui Login.class

Recovered code:

public boolean check(String password) {
return password.equals("secret");
}

Java is friendlier because:

  • Control flow is visible
  • Methods are named
  • Strings are easy to find

Reverse Engineering .NET ‚Äì The "Peek Through Glass"

.NET binaries (C#, VB.NET) compile to MSIL (Intermediate Language).

Example C# Code
C#
public class Auth {
public bool Login(string pwd) {
return pwd == "1234";
}
}

Tools: dnSpy, ILSpy

Decompiled code is almost perfect:

public bool Login(string pwd) {
return pwd == "1234";
}
.NET is one of the easiest languages to reverse engineer because:
‚Ä¢ Metadata is intact
‚Ä¢ Names are preserved
‚Ä¢ Control flow is readable

Mental Model #1: Language Dictates Approach

Your approach depends on language:

  • C/C++ ‚Üí low-level, manual memory, patterns
  • Java/.NET ‚Üí bytecode, readable structures
  • Python ‚Üí transparent, high-level logic
  • Android ‚Üí hybrid, multi-language

Cross-Language Patterns:

Regardless of language:

  • Functions ‚Üí identifiable via calls/offsets
  • Strings ‚Üí hints at logic
  • Loops and conditions ‚Üí control flow
  • Memory ‚Üí stack/heap/objects
Reverse engineering is language-agnostic at the core.

Example Workflow: Multi-Language RE

1. Identify language: PE/ELF/JAR/APK
2. Choose appropriate tool:
‚Ä¢ C/C++ ‚Üí Ghidra/IDA
‚Ä¢ Java ‚Üí Jadx
‚Ä¢ .NET ‚Üí dnSpy
‚Ä¢ Python ‚Üí uncompyle6
3. Explore sections/classes
4. Follow strings and functions
5. Map logic and reconstruct flow

Hands-On Example: Simple Cross-Language Analysis

C++ logic in native library for Android APK:

int multiply(int a, int b) {
return a * b;
}

Decompiled from .so ‚Üí see offsets and instructions

Reconstruct function ‚Üí verify input/output in JNI wrapper

Java Wrapper:

public class Calc {
static {
System.loadLibrary("nativeLib");
}
public native int multiply(int a, int b);
}
You now see multi-layered logic: C++ native function called from Java.
