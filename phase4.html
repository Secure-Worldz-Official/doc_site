The Archaeology of Code

PHASE 4: DECOMPILATION

From Fragments to Understanding

Now that you've mastered static analysis, it's time for one of the most exciting phases: decompilation.

Decompilation is like archaeology. You have fragments of an ancient civilization (the binary), and your job is to reconstruct the original code and logic as accurately as possible.

Unlike static analysis, decompilation tries to give you high-level code, often resembling the original C, C++, Java, or Python. But remember: it's never perfect.

What Decompilation Really Is

At its core, decompilation is:

  • Taking machine code or bytecode
  • Reconstructing readable code
  • Identifying structures, loops, and functions
Assembly Input
cmp eax, 1337
jne fail
call success
fail:
mov eax, 0
ret
Decompiled Output (C-like)
if (input == 1337) {
success();
} else {
return 0;
}
The logic is clear, even if variable names and comments are lost.

Decompiling C/C++ Binaries

C and C++ are compiled to raw machine code, often heavily optimized.

Original C Code
int sum(int a, int b) {
return a + b;
}
Compiled Assembly
mov eax, edi
add eax, esi
ret
Decompiled Output
int sum(int a, int b) {
return a + b;
}

Decompilers try to reconstruct high-level code, but face challenges:

  • Optimized instructions
  • Removed variable names
  • Inline functions
The mental skill is recognizing patterns.

Decompiling Different Languages

Java / .NET

Compile to bytecode, retain more structure

if (role.equals("admin")) {
unlockAll();
}

Decompiled via Jadx often matches almost perfectly

Python

Interpreted, .pyc files contain bytecode

uncompyle6 file.pyc > file.py

You recover high-level code, variable names, and structures

Bytecode languages are friendlier for reverse engineers. Python is often called "the glass house" â€” anyone can peek inside.

How to Read Decompiled Code

Decompiled code is rarely perfect:

  • Loops might appear as goto or while(true)
  • Structures may be replaced by arrays or pointers
  • Variable names are generic: v1, v2, a1, a2
Original C++
class User {
int id;
bool isAdmin;
public:
void login();
};
After Decompilation
struct User {
int v1;
char v2;
};
void User_login(User* this);
Even though names are gone, you recognize structure and methods.

Cross-Referencing Strings

Strings help in identifying function purposes:

Assembly
mov rdi, offset "Invalid license"
call printf
Decompiled Output
printf("Invalid license");
By following string references, you understand the logic behind functions.

Decompiled Control Flow

Decompilers attempt to rebuild loops and conditionals.

Assembly Snippet
cmp eax, 10
jl less
jmp end
less:
sub eax, 1
end:
Decompiled Output
if (eax < 10) {
eax = eax - 1;
}
Your job: verify if decompiler output matches real execution, especially with optimized binaries.

Android Decompilation

APK contains .dex bytecode. Using Jadx:

Terminal command
1jadx-gui classes.dex

You can explore:

  • Functions
  • Strings
  • Class hierarchies
  • Permissions usage
Even obfuscated apps often yield enough structure to understand flow.

Native Libraries in Android

Many apps include .so files (C/C++ libraries):

1. Use Ghidra or IDA to disassemble
2. Identify exported functions
3. Map to Java/Kotlin calls via JNI

Example JNI method:

JNI Native Function
JNIEXPORT jint JNICALL
Java_com_app_Main_add(JNIEnv* env,
jobject obj,
jint a,
jint b) {
return a + b;
}
You can reconstruct logic even across languages.

Limitations of Decompilation

Decompilers are helpers, not truth machines.

  • Optimizations: Compiler may remove or reorder code
  • Obfuscation: Names, classes, and logic flow may be hidden
  • Inline functions: Harder to separate in decompiled output
  • Pointers & memory access: May not translate cleanly

Example Workflows

Decompiling a Binary
1. Load binary into Ghidra
2. Analyze all sections
3. Identify .text section (code)
4. Look for string references
5. Decompile functions
6. Cross-reference calls and data

This is your daily reverse engineering routine.

Android APK
1. Unzip APK
2. Use Jadx to view .dex files
3. Identify key classes and methods
4. Analyze string constants
5. Map functions to app behavior

Mental Model #2: Guesswork With Structure

You are guessing intelligently. Every piece of code you see is a hint. Every string or function call gives context.
