What Is a Binary File, Really?

When people say "binary," they imagine something mysterious. In reality, a binary file is just a structured sequence of bytes.

That's it.

An executable file is not special because it is executable. It is special because the operating system understands its structure.

A binary answers questions like:

  • Where does execution start?
  • Which parts are code?
  • Which parts are data?
  • Which libraries are needed?

Without structure, the OS cannot run anything.

Common Binary Formats You Will See

Different operating systems use different binary formats.

Windows – PE
  • .exe
  • .dll
Linux – ELF
  • Most Linux executables
macOS – Mach-O
  • macOS binaries
Android – APK
  • Container, not pure binary
  • Contains DEX, native libraries, resources
As a reverse engineer, you do not memorize formats — you recognize patterns.

Inside a Binary: High-Level View

Every executable has sections.

Common sections include:

Section Purpose
Code section Instructions
Data section Global/static data
Read-only data Strings, constants
Import table External functions
Export table Functions provided

When you open a binary in Ghidra or IDA, you are not "hacking." You are reading a structured file.

Let's Touch a Real Binary (ELF Example)

On Linux:

Terminal command
1file app

Output:

Example output
1ELF 64-bit LSB executable

This already tells you:

  • Architecture (64-bit)
  • Endianness
  • Platform

Now try:

Terminal command
1readelf -h app

You will see:

  • Entry point
  • Section offsets
  • Header information
This is reverse engineering. Understanding structure before logic.

Mental Model #1: A Binary Is a Map, Not a Wall

A binary is not a locked door. It is a map.

If you know how to read maps, you can navigate any city.

CPU Basics: The Brain of the Program

A CPU does not execute code randomly.

It follows a simple cycle:

  • Fetch instruction
  • Decode instruction
  • Execute instruction
  • Move to next instruction
Reverse engineering is following this flow without panic.

Registers: The CPU's Working Desk

Registers are tiny, ultra-fast storage locations inside the CPU.

On x86-64, common registers include:

RAX
accumulator
RBX
base
RCX
counter
RDX
data
RSP
stack pointer
RIP
instruction pointer
If memory is a warehouse, registers are the items currently in your hands.

Memory: Where Everything Lives

Programs do not live "inside the CPU." They live in memory.

Memory is divided conceptually into regions:

Code Segment
• Instructions
Stack
• Function calls
• Local variables
• Return addresses
Heap
• Dynamically allocated memory

Understanding this layout is essential.

Stack: The Backbone of Execution

The stack follows a simple rule: Last In, First Out

When a function is called:

  • Arguments are placed
  • Return address is saved
  • Local variables are created

When a function returns:

  • Stack is cleaned
  • Control goes back
This is why stack analysis is powerful.

Calling Conventions: The Rules of Conversation

Calling conventions define:

  • How arguments are passed
  • Which registers are used
  • Who cleans the stack

Example (x86-64 System V):

  • Arguments passed via registers
  • Return value in RAX
Reverse engineering becomes easier when you know the rules.

Example: Simple Function Call in Assembly

C code:

C code
1int add(int a, int b) {
2 return a + b;
3}

Assembly (simplified):

Assembly
1mov eax, edi
2add eax, esi
3ret
No magic. Just data movement and addition.

Control Flow: How Programs Decide

Every decision boils down to:

  • Compare
  • Jump

Example:

Assembly decision
1cmp eax, 10
2jl less
This single instruction pair decides a path.

Reverse engineering is learning to follow paths, not instructions.

Mental Model #2: Ignore Syntax, Follow Flow

Beginners stare at instructions. Experts follow execution paths.

Always ask:

  • Where does execution go next?
  • Why did it choose this path?

Introduction to Disassemblers (No Worship)

Tools like:

  • Ghidra
  • IDA
  • objdump

do one thing well: They translate bytes into readable instructions.

They do not understand logic. They show structure. You supply meaning.

Opening a Binary in Ghidra (Conceptually)

When you load a binary:

  • Ghidra parses headers
  • Identifies sections
  • Finds entry point
  • Disassembles code
You are not "breaking" anything. You are reading.

Strings, Again (But Deeper)

Strings often reveal:

  • Error paths
  • Debug messages
  • Hidden features

In binaries, strings often sit near code that uses them.

That relationship matters.

Mental Model #3: Code and Data Are Neighbors

Instructions reference data. Data hints at logic.

Follow references.

Android APKs: A Special Case

An APK is a ZIP file.

Inside:

  • classes.dex (bytecode)
  • lib/ (native libraries)
  • AndroidManifest.xml
  • resources
Reverse engineering Android starts by unpacking, not disassembling.

DEX and Bytecode (High-Level)

DEX is not machine code. It is bytecode for the Android runtime.

This makes Android reverse engineering more readable than native binaries.

Native Code Inside Apps

Many apps use:

  • C
  • C++

These live as .so files.

Once inside, they behave like normal ELF binaries. Same rules apply.

Understand:
