PHASE 3: STATIC ANALYSIS

Opening the First Major Toolkit

Alright, now that you know what a binary really is and how CPUs think, it's time to open the first major toolkit: static analysis.

Static analysis is like reading a diary without knocking on the door. You never run the program. You just listen to what it tells you through structure, strings, and instructions.

The goal here is simple:

  • Learn about the program's logic
  • See how it handles data
  • Predict behavior without touching the execution

Why Static Analysis Matters

Why bother not running the program?

Because sometimes running it is:

  • Dangerous (malware, unknown binaries)
  • Irreproducible (requires specific environment)
  • Too slow for understanding the big picture

Static analysis allows you to:

Peek at Control Flow

See how the program makes decisions without executing it

Discover Strings

Find error messages, debug logs, API endpoints

Understand Function Calls

Map out how functions interact with each other

Find Dependencies

Identify libraries and external resources the program needs

First Tool: strings

You saw strings in Phase 2, but now we use it intentionally.

Terminal command
1strings program.bin | less

Look for:

  • Error messages
  • Debug logs
  • API endpoints
  • Licensing or trial messages

Example output:

Example output
1"Enter password:"
2"License expired"
3"Connecting to update server"

Even without execution, you know:

  • Program asks for a password
  • Program has licensing logic
  • Network communication exists

You now predict behavior. That's the essence of static analysis.

Inspecting Headers and Metadata

Every binary contains headers with important information.

Linux ELF example:

Terminal commands
1readelf -h program.bin
2readelf -S program.bin

-h shows:

  • Architecture (32/64-bit)
  • Entry point
  • Endianness

-S shows:

  • Sections
  • Offsets
  • Sizes

Windows PE example:

Windows command
1dumpbin /headers program.exe

You see:

  • Entry point
  • Sections: .text, .data, .rdata
  • Imported DLLs
This is program anatomy.

Disassembly Basics

Now we step slightly deeper.

Disassemblers like Ghidra or IDA Pro turn raw bytes into readable assembly:

Assembly instructions
1cmp eax, 5
2jne fail
3call success

Here we see:

  • Comparison
  • Conditional jump
  • Function call
Notice: no variable names, no comments. Only raw logic.

Understanding Control Flow

Control flow defines the program's decisions.

Key instructions:

  • jmp — unconditional jump
  • je/jne/jg/jl — conditional jumps
  • call — function call
  • ret — function return
C Code
if (x == 42) {
unlock_feature();
} else {
deny_access();
}
Assembly Version
cmp eax, 42
jne deny
call unlock_feature
jmp end
deny:
call deny_access
end:
Static analysis is mapping these paths without running them.

Sections and Their Meaning

A binary's sections tell you what type of data lives where.

Section Purpose
.text Executable code
.data Global/static data
.rodata Read-only constants, strings
.bss Uninitialized global/static data
.rdata (Windows) Read-only resources
Knowing section contents helps you locate important info fast.

Reversing Functions

Once disassembled, we start function analysis.

Look at:

  • Function start and end
  • Arguments (from calling conventions)
  • Registers used
  • Local variables (stack offsets)
C Function
int add(int a, int b) {
return a + b;
}
Disassembly (simplified)
mov eax, edi
add eax, esi
ret

Registers:

  • edi → first argument a
  • esi → second argument b
  • eax → return value
Even without running, we know the function adds two numbers.

String References in Assembly

Strings often reside in .rodata or .rdata.

Example:

Assembly with string reference
1mov rdi, offset error_msg
2call printf
  • error_msg → could be "Invalid input"
  • Assembly shows where and how messages are used
This is critical for understanding program behavior without execution.

Finding Imports

Imports tell you what external functions the program relies on.

Windows (PE)
dumpbin /imports program.exe

You might see:

  • MessageBoxA → GUI message box
  • CreateFileA → file operations
  • Connect → network call
Linux (ELF)
ldd program

Shows linked shared libraries:

  • libc.so
  • libssl.so

Gives context for runtime behavior

Mental Model #1: Everything Has a Trail

Even without running a program:

Strings leak logic
Error messages reveal what the program checks for
Imports show capabilities
External functions tell what the program can do
Sections hint at structure
How the program organizes its code and data
Jumps reveal control flow
Conditional branches show decision points
Static analysis is detective work: piece together the story from clues.

Using Ghidra for Static Analysis

Ghidra simplifies static analysis:

  1. Load binary
  2. Auto-analyze (Ghidra parses headers, sections, functions)
  3. Explore .text for functions
  4. Look at disassembled instructions
  5. Check string references (Shift+F12)

Example: guessing license check logic

C logic (what we're looking for)
if (license_code == 12345) {
enable_features();
} else {
disable_features();
}

In Ghidra, you see:

  • Comparison instructions
  • Conditional jumps
  • Function call to enable/disable features
You now know logic without execution.

Static Analysis of Android APKs

APK is a ZIP container:

Terminal command
1unzip app.apk -d unpacked/

Inside:

  • classes.dex → Java/Kotlin bytecode
  • lib/ → native libraries (.so)
  • AndroidManifest.xml → app permissions and activities

Use jadx:

Using jadx
1jadx-gui classes.dex
  • Explore Java code
  • Look for strings, conditionals, and functions
  • Analyze logic statically

Dealing With Obfuscation

Some binaries intentionally hide logic:

  • Names replaced with meaningless letters
  • Control flow flattened
  • Junk instructions added
Static analysis helps you see structure first, so later dynamic analysis is easier.

Mental Model #2: Start Wide, Then Zoom

1. Look at the whole binary
Sections, imports, strings - get the big picture
2. Identify critical functions
What looks important? Entry points, main functions
3. Explore instructions
Control flow, data usage, and logic patterns
Static analysis is layered observation, not hacking.
