What Binary Patching Really Means

Binary patching means:

  • Changing instructions inside a compiled program
  • Modifying control flow
  • Altering logic checks
  • Redirecting execution paths
Think of software like a railway system ðŸš†
Binary patching is changing the tracks, not the train.

You are not destroying the program â€” you are redirecting its decisions.

Why Binary Patching Exists

Binary patching is used for:

Understanding
  • Understanding program logic
  • Research and education
  • Confirming hypotheses
Practical
  • Fixing bugs without source
  • Security testing
  • Malware analysis

Most importantly:

Binary patching teaches how fragile logic really is.

One instruction. One jump. Everything changes.

The Golden Rule Before Touching Anything

Before patching, always:

  • Understand the logic (static analysis)
  • Confirm behavior (dynamic analysis)
  • Identify the exact instruction responsible
Patching without understanding is guessing.
Reverse engineering is not guessing.

Control Flow: The Heart of Patching

Most patches revolve around control flow instructions.

Common x86/x64 Control Flow Instructions
Assembly
je â†’ jump if equal
2jne â†’ jump if not equal
3jg â†’ jump if greater
4jl â†’ jump if less
5jmp â†’ unconditional jump

Example assembly:

Original Logic
Assembly
1cmp eax, 1
2jne fail
3call success

Logic: If eax != 1, go to fail. Else, call success.

A single instruction controls everything.

The Simplest Patch: Conditional to Unconditional

Original:

1cmp eax, 1
2jne fail

Patched (Option 1 - Always succeed):

1cmp eax, 1
2jmp success

Patched (Option 2 - Remove check):

1cmp eax, 1
2nop
3nop
Now the check exists, but it does nothing.

Static Patching vs Runtime Patching

Static Patching
  • Modify the binary file on disk
  • Changes are permanent
  • Requires recalculating offsets

Editing the blueprint permanently

Runtime Patching
  • Modify instructions in memory
  • Changes disappear after restart
  • Safer for experimentation

Changing the tracks while the train runs

Both are important. Runtime patching is usually learned first.

Runtime Patching Example (x64dbg)

Imagine a password check inside a Windows program.

1. Load binary in x64dbg
2. Run until password check
3. Find comparison instruction
4. Right-click â†’ Assemble

Patching with gdb (Linux)

Example:

1cmp eax, 1337
2jne fail

At runtime in gdb:

gdb Commands
(gdb) set $eax = 1337
(gdb) continue

Or patch instruction bytes:

(gdb) set {unsigned char}0x40123A = 0x90
(gdb) set {unsigned char}0x40123B = 0x90
Two bytes. Two nops. Logic gone.

Mental Models for Binary Patching

Mental Model #1: Patching Is Proof, Not Cheating

Binary patching proves:

  • You understood the logic
  • You identified the correct instruction
  • You confirmed behavior dynamically
If your patch works, your understanding was correct.

Mental Model #2: Small Changes, Big Effects

Reverse engineering teaches humility.

A billion-line software product can fall because of one jump instruction.

This is why defense matters, layers matter, server-side validation matters.

Practice Ideas

  • Patch a trial expiration check
  • Patch a debug-only feature
  • Patch a hidden menu unlock
  • Patch a fake license check (self-written program)
Always practice on your own binaries first.

Common Beginner Mistakes

  • Patching before understanding
  • Trusting decompiler blindly
  • Forgetting instruction size
  • Breaking offsets and crashes
Good reverse engineers go slow.
