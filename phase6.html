Shadowing Software in Motion

PHASE 6: DYNAMIC ANALYSIS

The Most Interactive Part

Welcome to the most interactive part of reverse engineering! Static analysis taught you how to read programs silently, and decompilation helped you reconstruct logic. Now it's time to watch software in motion and understand its behavior in real-time.

Dynamic analysis is like shadowing a person instead of reading their diary. You see exactly what happens when the program runs.

Why Dynamic Analysis Matters

Programs aren't just static bytes; they:

  • Allocate memory
  • Call APIs
  • Communicate over networks
  • Perform conditional actions
Static analysis gives hints, but running the program reveals reality.

Benefits of dynamic analysis:

Observe Runtime

See actual execution flow in real-time

Detect Hidden Logic

Find code not visible in static analysis

Understand Security

Watch encryption, password checks, and loops

Safe Malware Analysis

Debug malware safely in sandbox environments

Debuggers: Your Best Friends

A debugger lets you pause, step through, and inspect program execution.

Example: gdb with a C Program

C Program
C
#include <stdio.h>
int main() {
int secret = 42;
int input;
printf("Guess the number: ");
scanf("%d", &input);
if(input == secret) {
printf("Correct!\n");
} else {
printf("Try again.\n");
}
return 0;
}

Run under gdb:

gdb ./guess

Common debugger commands:

break main

Set breakpoint at main function

run

Start program execution

next

Step line by line (step over functions)

print secret

Inspect variable value

You can now see the value of secret in memory without changing code.

Breakpoints and Stepping

Breakpoints let you stop execution at a specific point.

Types of breakpoints:

  • Conditional breakpoints: stop only if a condition is met
  • Memory breakpoints: stop when a memory location is accessed
  • Function breakpoints: stop at function entry/exit

Step commands:

step ‚Üí step into function (enter the function)
next ‚Üí step over function (execute function as one step)
continue ‚Üí resume execution until next breakpoint

Dynamic Analysis Workflow

1. Identify functions of interest from static analysis
2. Set breakpoints or hooks
3. Observe variable values, memory, registers
4. Step through execution
5. Modify values for testing behavior
6. Record patterns for later patching or understanding

Runtime Behavior Patterns

Common patterns to watch for during dynamic analysis:

Password Checks

Compare user input with stored value

Encryption/Decryption

Watch keys in memory

API Calls

Observe network or file operations

Conditional Logic

Follow if/else branches in real-time

Mental Model #1: Think Like the Program

Predict what should happen
Watch if it matches your prediction
Step into instructions to understand deviations
Dynamic analysis trains intuition about execution flow.

Example: Reverse Engineering a Password Check

C code:

if(password[0] == 'A' && password[1] == 'B')
unlock();

Dynamic analysis approach:

1. Set breakpoint at unlock() function
2. Observe password memory values
3. Modify register/memory to test different paths
You now understand exact conditions needed for success.

Mental Model #2: Observation Before Modification

Observe first ‚Üí understand behavior
Modify second ‚Üí confirm logic
Never assume static analysis tells the full story
Dynamic analysis complements static analysis perfectly.
